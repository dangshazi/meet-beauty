## 2. 目标说明
这份任务清单面向 MVP 分阶段开发，目标不是一次性做完整产品，而是按“能跑通主流程、能验证假设、能稳定演示”的顺序推进。

核心主线：
- 先打通相机 + 人脸识别 + AR 渲染基础链路
- 再打通分析推荐 + 分步骤教学
- 最后补足轻量反馈、可观测性和上线准备

## 3. 开发阶段规划

### Phase 0：产品定义与技术预研
目标：确认 MVP 边界、验证核心技术可行性、建立开发骨架。

交付结果：
- 确认 MVP 页面流和功能范围
- 完成 Flutter 工程初始化
- 验证相机预览、Face Mesh、基础叠色效果可运行
- 输出技术选型结论

任务清单：
- 产品
- 明确首版只做口红和腮红教学
- 明确首版只做规则推荐，不做训练模型
- 明确首版页面为：首页、分析推荐页、相机教学页、结果页
- 设计
- 产出低保真页面流转图
- 定义首页、推荐页、教学页、结果页的文案框架
- 确定教学态 UI 样式，包括步骤提示、区域高亮、按钮布局
- 客户端
- 初始化 Flutter 项目结构
- 搭建基础路由、状态管理、主题配置
- 接入相机插件并完成实时预览 Demo
- 调研 MediaPipe 在 Flutter 中的接入方式
- 算法/CV
- 验证 Face Mesh 能稳定输出关键点
- 验证关键点驱动的唇部 polygon 绘制
- 验证脸颊区域 mask 的简化算法
- 验证口红和腮红颜色叠加效果
- QA
- 定义首轮技术验收标准
- 记录不同机型下的性能和发热观察项

阶段验收标准：
- 能在测试机上打开相机并看到实时画面
- 能获取 Face Mesh 关键点数据
- 能在唇部区域叠加基础颜色
- 团队已对 MVP 范围和技术路径达成一致

### Phase 1：核心视觉链路开发
目标：实现“相机 -> 关键点 -> 区域 mask -> 基础 AR 渲染”的稳定链路。

交付结果：
- 实时人脸检测和关键点追踪可用
- 口红与腮红 AR 效果可显示
- 教学页具备稳定的相机渲染能力

任务清单：
- 客户端
- 封装 CameraService，统一相机初始化、权限和帧流控制
- 封装 FaceTrackingService，统一关键点解析与异常处理
- 完成教学页的相机预览层和 overlay 层分离
- 实现基础 loading、权限拒绝、识别失败等状态 UI
- 算法/CV
- 定义唇部关键点集合与 polygon 顺序
- 定义左右脸颊区域计算规则
- 实现区域平滑、抖动过滤和关键点插值
- 实现颜色叠加透明度控制
- 实现区域边界裁切，避免明显溢出
- 前端/UI
- 完成教学页视觉样式
- 完成步骤卡片、操作按钮、提示文案 UI
- 调整 overlay 的层级关系，保证遮罩和文案可读
- QA
- 测试正脸、轻微侧脸、弱光下的稳定性
- 测试相机切前后台时的资源释放与恢复
- 测试权限拒绝后的回退逻辑

阶段验收标准：
- 唇部和腮红区域可稳定跟随人脸移动
- 效果刷新没有明显撕裂或大范围抖动
- 教学页可连续运行 3 分钟以上而不崩溃

### Phase 2：分析推荐流程开发
目标：实现“拍脸分析 -> 标签结果 -> 妆容推荐 -> 进入教学”的产品闭环前半段。

交付结果：
- 面部分析页可输出基础结果
- 推荐页可给出妆容建议与理由
- 用户可以从推荐页进入教学页

任务清单：
- 产品/内容
- 定义首版推荐维度：脸型、肤色、唇形厚薄
- 定义 2-4 套首版固定妆容模板
- 编写推荐理由文案与风险表述文案
- 客户端
- 开发分析状态页与推荐结果页
- 建立 AnalysisResult、MakeupProfile、Recommendation 数据模型
- 实现拍脸后分析 loading、成功、失败三种状态
- 实现推荐结果展示和“开始学习”交互
- 算法/CV
- 实现基于 landmarks 的脸长/脸宽比例计算
- 实现唇厚比例的简单估算
- 实现肤色基础分类的轻量规则
- 实现规则引擎，将面部特征映射到妆容模板
- 数据/配置
- 建立本地妆容模板配置文件
- 建立规则映射表，支持后续调整阈值
- QA
- 测试不同用户照片下的推荐稳定性
- 测试识别失败时的回退和跳过分析能力
- 验证推荐结果始终能返回可用模板

阶段验收标准：
- 用户可完成一次完整的拍脸分析流程
- 每次分析都能输出可解释的推荐结果
- 推荐页能稳定进入教学流程

### Phase 3：分步骤教学流程开发
目标：实现真正体现差异化的“AI 教学”主流程。

交付结果：
- 教学步骤控制器可驱动整个教程流程
- 口红、腮红两步教学可以顺序完成
- 用户可以按自己的节奏完成练习

任务清单：
- 产品/内容
- 定义每个步骤的教学文案
- 定义步骤完成条件和可跳过条件
- 定义重练、返回、切换模板的行为逻辑
- 客户端
- 建立 TutorialSession、TutorialStep、TutorialProgress 数据结构
- 实现步骤状态机：未开始、进行中、完成
- 实现“下一步”“重试本步”“结束教学”等交互
- 实现步骤级提示卡和动画反馈
- 算法/CV
- 将当前步骤与具体 overlay 区域绑定
- 为不同妆容模板注入不同颜色和提示参数
- 预留步骤校验接口，后续接规则评分
- UI/交互
- 优化单手操作体验
- 处理文案与相机画面遮挡冲突
- 增加步骤过渡态与完成态提示
- QA
- 测试中断恢复，例如来电、切后台、权限变化
- 测试教学流程回退、重试和重复进入
- 测试不同妆容模板下的步骤显示是否正确

阶段验收标准：
- 用户可以顺序完成口红和腮红教学
- 教学过程中步骤信息始终正确
- 重试和下一步逻辑无错乱

### Phase 4：基础反馈与评分开发
目标：为教学流程补足轻量反馈，让 MVP 更像“教练”而不是“滤镜”。

交付结果：
- 结果页可展示评分和建议
- 系统可以输出基础纠错结果
- 用户可再练一次或切换模板

任务清单：
- 产品/内容
- 定义评分规则和反馈文案池
- 明确反馈只针对操作结果，不评价用户外貌
- 定义结果页结构和再练路径
- 客户端
- 开发结果页 UI
- 实现评分结果展示、建议展示、再练一次入口
- 实现返回推荐页或首页的跳转逻辑
- 算法/CV
- 实现覆盖率估算
- 实现边界越界检测
- 实现步骤完成度计算
- 基于规则结果输出简单反馈标签
- QA
- 验证评分逻辑在边界情况下不输出明显错误反馈
- 验证没有数据时的降级方案
- 验证结果页跳转逻辑完整

阶段验收标准：
- 完成教学后必定能看到结果页
- 评分逻辑可输出基础可解释结果
- 用户可以无障碍进入下一轮练习

### Phase 5：稳定性优化与 MVP 上线准备
目标：从“能演示”提升到“可测试、可试用、可对外展示”。

交付结果：
- 基础埋点与日志能力
- 稳定性和性能优化
- 首版测试清单与发布包

任务清单：
- 客户端
- 优化关键帧处理频率，降低卡顿与发热
- 优化页面切换和资源释放
- 增加崩溃保护和兜底提示
- 埋点/数据
- 增加关键漏斗埋点：开始分析、分析成功、进入教学、完成教学、再练一次
- 增加错误日志：相机初始化失败、关键点解析失败、推荐失败
- 测试
- 做一轮冒烟测试
- 做一轮真机兼容测试
- 做一轮性能测试，包括 FPS、内存、发热
- 运营/产品
- 准备演示脚本
- 准备首批用户访谈问题
- 准备隐私说明和权限说明

阶段验收标准：
- 主流程 crash rate 可接受
- 首轮对外试用版本可完整安装和体验
- 已具备基础用户验证能力

## 5. 团队分工建议

- 管理 PRD、优先级、功能边界
- 定义妆容模板和推荐逻辑规则
- 编写关键页面文案与用户访谈提纲

- 负责首页、推荐页、教学页、结果页设计
- 定义教学提示组件、状态反馈组件、空状态页面
- 跟进相机页实际遮挡和可读性问题

### Flutter 工程师
- 负责页面开发、状态管理、相机接入、交互实现
- 负责服务封装、路由、埋点和性能优化

### CV/算法工程师
- 负责 Face Mesh 接入、特征规则、区域计算、反馈规则
- 输出阈值配置和算法说明

### QA
- 负责主流程测试、机型兼容、弱网与权限异常测试
- 跟踪稳定性与回归质量

## 6. 任务优先级
### P0
- 相机接入
- Face Mesh 接入
- 唇部和脸颊 AR 区域渲染
- 分析结果输出
- 推荐页
- 教学步骤控制
- 结果页

### P1
- 基础纠错
- 模板切换
- 埋点
- 异常兜底

### P2
- 更多模板
- 更细推荐逻辑
- 更丰富反馈表达

## 7. 风险提示
- 如果 Phase 1 的相机与关键点链路不稳定，后续阶段全部会被拖慢，因此必须优先解决。
- 如果规则推荐效果不理想，也不要阻塞教学主流程，推荐模块要支持快速降级为固定模板。
- 如果评分误判太多，宁可弱化反馈，也不要输出错误且打击用户的结论。

## 8. 最终上线门槛
满足以下条件即可视为 MVP 可上线测试：
- 用户能在 1 分钟内完成分析并进入教学
- 用户能顺利完成口红和腮红两个步骤
- App 在目标机型上主流程基本稳定
- 推荐、教学、反馈三段链路全部打通
- 有基础埋点和错误日志，便于后续迭代
